# Concert SBOM Upload

Concert is internal-network only — GitHub Actions can't POST to it directly. The pipeline generates SBOMs automatically; you download and upload by hand.

## Pipeline artifacts (7-day retention)

| Artifact | Concert destination | Source |
|----------|---------------------|--------|
| `sbom-image-<svc>-<tag>` | Dimensions → Software composition → CycloneDX SBOMs | Image |
| `sbom-code-<svc>-<tag>` | Dimensions → Software composition → CycloneDX SBOMs | Source code |
| `sbom-app-<tag>` | Inventory → Application inventory → Define application → From SBOM | — |
| `sbom-all-<tag>.zip` | All of the above bundled | — |

Generated by [`docker-build.yml`](../../.github/workflows/docker-build.yml): `code-scan` (cdxgen), `sbom` (Syft image-scan), `app-sbom` (ConcertDef), `sbom-bundle` (zip).

## Upload steps

> ⚠️ **Order matters.** The app SBOM (ConcertDef) **must** be uploaded first. Concert uses it
> to register the application and declare which container images belong to it. Image and code
> SBOMs uploaded before the app SBOM will fail with:
> *"The application and the component relation is not existing to make the co-relation of new
> build artifact instance."*

1. **Actions** → `Docker Build & Push` run → **Artifacts** → download `sbom-all-<tag>.zip`
2. **App SBOM first** — Inventory → Application inventory → **Define application → From SBOM**
   → upload `banking-demo-<tag>-app.concertdef.json`
   - Wait for the upload to complete and the application to appear in the inventory before
     proceeding.
3. **Image SBOMs** — for each `<svc>-<tag>.cdx.json`:
   - Dimensions → Software composition → CycloneDX SBOMs → **Upload**
   - File type: `Package (CycloneDX)` · Source: `Image`
4. **Code SBOMs** — same flow, Source: `Source code`

> Upload one file at a time — Concert doesn't handle concurrent uploads.

## Troubleshooting

**"The application and the component relation is not existing…"** — Two possible causes:

1. **Wrong upload order** — Image or code SBOM uploaded before the app SBOM. Upload
   `banking-demo-<tag>-app.concertdef.json` first (step 2), then re-upload the rejected files.

2. **Image/repo name mismatch** — `metadata.component.name` in the SBOM does not exactly match
   the corresponding `name` field in the app SBOM blueprint:
   - Image SBOMs: must match `blueprint-container.name` = bare image path, e.g.
     `ghcr.io/dungxnd/banking-demo-revamp/account-service` (no tag, lowercase).
   - Code SBOMs: must match `blueprint-code.name` = repo name = `banking-demo`.
     The `externalReferences[vcs].url` must also equal `blueprint-code.purl` (repo root URL,
     no sub-path). A sub-path `website` reference in the code SBOM causes
     *"co-relation of new repository instance"* — fixed in the workflow (sub-path removed).

**SBOM shows "Uncorrelated"** — Application Perspective must filter by both
`kubernetes.cluster.name = banking-dung` and `kubernetes.namespace.name = banking`.
See [`09-pod-service-detection.md`](./09-pod-service-detection.md).
