OpenTelemetry Tracing

Giải thích Trace, Span, Context Propagation — và bug trong banking-demo

1. Các định nghĩa cốt lõi

Khái niệm Định nghĩa Ví dụ trong banking-demo
Trace Toàn bộ hành trình của 1 request từ đầu đến cuối, qua bao nhiêu service cũng được. Mỗi trace có 1 traceId duy nhất. User POST /api/transfers → đi qua Kong → api-producer → transfer-service: đó là 1 trace
Span Một đơn vị công việc trong trace. Có spanId, tên, thời gian bắt đầu/kết thúc, attributes, và parentSpanId. Spans lồng nhau tạo thành cây. rpc.request (api-producer), nats.handler (transfer-service)
SpanContext Phần "định danh" của span: traceId + spanId + flags. Cái này được truyền qua mạng để các service khác biết mình thuộc về trace nào. traceparent: 00-abc123...-def456...-01
Context Propagation Cơ chế truyền SpanContext từ service này sang service kia qua header (HTTP) hoặc message header (NATS, Kafka). api-producer inject traceparent vào NATS header → transfer-service extract ra
Tracer Object dùng để tạo span. Mỗi service tạo 1 tracer với tên của mình. Không có tracer gọi Start() → không có span. otel.Tracer("transfer-service")
TracerProvider Factory tạo ra Tracer. Cần được cấu hình với Exporter (nơi gửi spans đến). Nếu không cấu hình → spans bị bỏ vào void. sdktrace.NewTracerProvider(WithBatcher(otlpExporter))
OTLP Exporter Gửi spans từ process Go đến collector (Instana agent) qua gRPC port 4317. Nếu OTEL_EXPORTER_OTLP_ENDPOINT không set → exporter không tồn tại. http://$(NODE_IP):4317 trong Helm values
Parent Span Span cha. Khi tạo span mới từ 1 context đã có span, span mới tự động trở thành con của span đang active trong context đó. rpc.request là cha của nats.handler
Remote Parent Span cha ở service khác. Được khôi phục qua Extract() — tạo một "link" xuyên process trong cùng 1 trace. api-producer inject → consumer extract → child span vẫn cùng traceId

2. Vòng đời của 1 Span

// 1. Lấy tracer (thường làm 1 lần lúc khởi động)
tracer := otel.Tracer("my-service")

// 2. Bắt đầu span — trả về context mới có chứa span này
ctx, span := tracer.Start(ctx, "operation.name",
    trace.WithSpanKind(trace.SpanKindServer),
    trace.WithAttributes(
        attribute.String("http.method", "POST"),
    ),
)
defer span.End()  // 3. Kết thúc span khi hàm return

// 4. Làm việc... đặt thêm attributes nếu cần
span.SetAttributes(attribute.Int("rpc.status", 200))

// 5. Đánh dấu lỗi nếu có
span.SetStatus(codes.Error, err.Error())

3. Context Propagation — Hoạt động như thế nào?

  1. Producer tạo span — tracer.Start(ctx, "rpc.request"). Lúc này ctx chứa spanId của span vừa tạo.
  2. Inject vào header — otel.GetTextMapPropagator().Inject(ctx, HeaderCarrier(hdr)). Lệnh này ghi SpanContext vào NATS header dưới dạng:
    traceparent: 00-{traceId}-{spanId}-01
  3. NATS giao message — header traceparent đi kèm message đến consumer.
  4. Consumer Extract — ctx = otel.GetTextMapPropagator().Extract(ctx, HeaderCarrier(headers)). Lệnh này đọc header và đặt SpanContext (của producer) vào ctx như một remote parent.
  5. Consumer tạo child span — ctx, span := tracer.Start(ctx, "nats.handler"). Vì ctx đang có remote parent, span mới tự động trở thành con — cùng traceId, parentSpanId = spanId của producer.
⚠ Quan trọng: Extract() chỉ đọc header vào context. Nó không tạo span. Nếu sau Extract không có tracer.Start() thì không có gì được gửi đến Instana cả.

4. traceparent Header — Anatomy

Format: traceparent: 00-{traceId}-{parentSpanId}-{flags}
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             ↑  ←————————— traceId (32 hex) ————————→ ←— spanId (16 hex) —→  ↑
          version                                                           sampled=true
  • traceId: ID duy nhất cho toàn bộ trace — giống nhau ở mọi service trong cùng 1 request
  • spanId (parentSpanId): ID của span cha — consumer dùng cái này để "link" span của mình vào đúng chỗ trong cây
  • flags: 01 = sampled (ghi lại), 00 = không ghi lại

6. Tại sao fix này áp dụng cho tất cả service?

Consumer là struct dùng chung trong internal/nats/consumer.go. Tất cả 4 service đều gọi internnats.NewConsumer(...) — không service nào có dispatcher riêng. Sửa 1 chỗ → áp dụng ngay cho cả 4:
Service Subject prefix Tracer name (otel.Tracer(name)) Span sẽ thấy
auth-service banking.auth "auth-service" nats.handler
account-service banking.account "account-service" nats.handler
transfer-service banking.transfer "transfer-service" nats.handler
notification-service banking.notification "notification-service" nats.handler

7. Tóm tắt quy trình đầy đủ

// ─── api-producer/rpc.go ───────────────────────────────────────────
ctx, span := tracer.Start(ctx, "rpc.request")   // Tạo span cha
defer span.End()
otel.GetTextMapPropagator().Inject(ctx, HeaderCarrier(hdr))
// → NATS header: traceparent: 00-{traceId}-{rpc.request spanId}-01
c.nc.RequestMsgWithContext(ctx, &nats.Msg{Header: hdr, ...})

// ─── internal/nats/consumer.go ─────────────────────────────────────
ctx = otel.GetTextMapPropagator().Extract(ctx, HeaderCarrier(req.Headers()))
// ctx bây giờ biết: remote parent = rpc.request spanId

ctx, span := c.tracer.Start(ctx, "nats.handler")   // Tạo span con ✅
// → parentSpanId = rpc.request spanId → cùng traceId → thấy trong Instana
defer span.End()
h(ctx, action, payload, headers)   // Handler chạy bên trong span này
