1. Overview & Architecture

This chart deploys the Instana Agent as a privileged DaemonSet on every schedulable node. Internally it creates a Kubernetes Operator (controller-manager) that reconciles an Agent CustomResource (CR) built from the Helm values. The CRD must be applied manually before upgrades (see §12).

Key Kubernetes Objects Created

ResourcePurpose
DaemonSet / instana-agentRuns agent on every node for APM + infra monitoring
Deployment / instana-agent-controller-managerOperator that reconciles the Agent CR
Deployment / instana-agent-k8sensorIsolated Kubernetes-sensor pod(s) — enabled via k8s_sensor.deployment.enabled: true
CRD / agents.instana.ioCustom resource definition; must be applied manually on upgrades
Secret / instana-agent-keyStores agent key / download key (or BYO via agent.keysSecret)
Service / instana-agentExposes OTel gRPC:4317, HTTP:4318, Prometheus remote-write inside cluster
ClusterRole + BindingRBAC for full cluster observability

2. Install / Upgrade Commands

Initial Install

kubectl create namespace instana-agent

helm install instana-agent \
  --repo https://agents.instana.io/helm \
  --namespace instana-agent \
  --set agent.key="${INSTANA_AGENT_KEY}" \
  --set agent.endpointHost="${INSTANA_ENDPOINT_HOST}" \
  --set zone.name="ZONE_NAME" \
  --set cluster.name="CLUSTER_NAME" \
  instana-agent

Upgrade (with CRD pre-apply — required!)

helm pull --repo https://agents.instana.io/helm --untar instana-agent
kubectl apply -f instana-agent/crds
helm upgrade instana-agent instana-agent \
  --namespace instana-agent \
  --repo https://agents.instana.io/helm \
  --reuse-values

Uninstall (manual CRD cleanup needed)

helm uninstall instana-agent -n instana-agent \
  && kubectl patch agent instana-agent -n instana-agent \
     -p '{"metadata":{"finalizers":null}}' --type=merge \
  && kubectl delete crd/agents.instana.io

OpenShift pre-requisites

oc adm policy add-scc-to-user privileged -z instana-agent
oc adm policy add-scc-to-user anyuid -z instana-agent-remote -n instana-agent

6. agent.env / agent.configuration_yaml

agent.env (legacy string-map)

Simple key=value map for environment variables. Superseded by agent.pod.env which supports full Kubernetes syntax but both still work.

agent:
  env:
    INSTANA_AGENT_TAGS: dev
    INSTANA_GIT_REMOTE_REPOSITORY: "https://github.com/org/repo.git"
    INSTANA_GIT_REMOTE_BRANCH: "main"
    # INSTANA_GIT_REMOTE_USERNAME: set via --set at install time
    INSTANA_GIT_REMOTE_PASSWORD: ""

agent.configuration_yaml — Inline sensor config (hot-reloaded)

Embeds the configuration.yaml content directly in the Helm values. The operator stores it in a Kubernetes Secret (not a ConfigMap), so plain-text passwords are never exposed after deployment. Changes require a helm upgrade OR can be hot-reloaded via the Instana Web API.

agent:
  configuration_yaml: |
    # Secrets redaction
    com.instana.secrets:
      matcher: 'contains-ignore-case'
      list:
        - 'key'
        - 'password'
        - 'secret'

    # Host tags
    com.instana.plugin.host:
      tags:
        - 'dev'
        - 'kubernetes'
Git-based config management (alternative): Set INSTANA_GIT_REMOTE_REPOSITORY / INSTANA_GIT_REMOTE_BRANCH / INSTANA_GIT_REMOTE_USERNAME env vars so the agent pulls instana/configuration.yaml from your Git repo on every startup — no helm upgrade needed for sensor config changes.

7. k8s_sensor.* — Kubernetes Sensor Deployment

When k8s_sensor.deployment.enabled: true, the Kubernetes sensor runs as a separate Deployment (HA, 3 replicas by default) instead of inside every DaemonSet pod. This reduces DaemonSet memory consumption significantly on large clusters.

11. Multi-Zone DaemonSets

Deploy separate DaemonSets per node pool using tolerations as the basis for zone association:

zones:
  - name: pool-01
    tolerations:
    - key: "pool"
      operator: "Equal"
      value: "pool-01"
      effect: "NoExecute"
  - name: pool-02
    tolerations:
    - key: "pool"
      operator: "Equal"
      value: "pool-02"
      effect: "NoExecute"

13. Upgrade Gotchas

CRDs are NOT auto-updated on helm upgrade. You must manually apply kubectl apply -f instana-agent/crds before every upgrade, especially when migrating from chart v1 → v2.
ScenarioAction
Sensor config change onlyGit push → trigger Web API hot-reload (no helm needed)
Helm values change (resources, zones, k8s_sensor etc.)helm upgrade --reuse-values -f values.yaml
Chart version upgradePre-apply CRDs, then helm upgrade
v1 → v2 migrationRun convert-agent-configmap-to-custom-values.sh first, then pre-apply CRDs
Manual ConfigMap edits (v1)Use convert-agent-configmap-to-custom-values.sh to extract to custom-values.yaml
Full uninstallHelm uninstall + patch finalizers + delete CRD (manual)
